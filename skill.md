# 경력기술서 자동 생성 지시문

## 트리거

사용자가 "오늘치 경력기술서 생성해줘" 또는 "오늘치 생성" 또는 "daily" 라고 입력하면 아래 절차를 그대로 실행한다.

---

## 실행 절차

### STEP 1: history.json 읽기

레포 루트의 `history.json`을 읽는다. 파일이 없으면 아래 초기값으로 생성한다.

```json
{
  "current_cycle": 1,
  "remaining_in_cycle": ["영업", "마케팅", "재무회계", "인사", "생산관리", "IT개발", "MD바잉", "제약영업", "품질관리", "물류SCM", "홍보"],
  "used_combos": [],
  "history": []
}
```

### STEP 2: 오늘 날짜로 이미 생성한 적 있는지 확인

`history` 배열에서 오늘 날짜(YYYY-MM-DD) 항목이 이미 있으면, 사용자에게 "오늘치는 이미 생성되었습니다. 다시 생성할까요?"라고 묻고 답을 기다린다. "응" 또는 "yes"면 해당 날짜 항목을 history에서 삭제하고 진행. 아니면 종료.

### STEP 3: 직무 3개 뽑기 (사이클 균등 분배)

- `remaining_in_cycle` 배열의 길이가 3 미만이면:
  - `current_cycle`을 1 증가
  - `remaining_in_cycle`을 11개 직무 전체로 리셋
- `remaining_in_cycle`에서 무작위로 3개 직무 선택 (Bash/Python의 random 사용 가능, 직접 추론 추첨도 가능)
- 선택된 3개 직무를 `remaining_in_cycle`에서 제거

### STEP 4: 각 직무별 (연차 + 이직 상황) 조합 결정

각 직무에 대해:

**연차 풀**: 6개월~1년차, 1~3년차, 4~6년차
**이직 상황 풀**: 같은직무이동, 같은직무+업계이동, 직무변동, 사기업→공기업, 공기업→사기업

두 축을 곱하면 직무당 15개 조합. 그 중 `used_combos`에 `"{직무}|{연차}|{상황}"` 형태로 이미 존재하지 않는 조합만 후보로 두고 무작위 선택.

15개 조합이 모두 used 상태인 직무가 있으면(거의 발생 안 함, 약 55일 이후), 해당 직무에 한해 used_combos에서 그 직무 항목들을 모두 제거하고 다시 추첨.

선택된 조합은 `used_combos`에 즉시 추가.

### STEP 5: 각 직무별 출력 양식 2개 결정

**양식 풀:**
- F1: 업무단위·간단형 (스킬의 패턴 1, 짧은 버전)
- F2: 업무단위·상세형 (스킬의 양식 A, 카테고리화 + HOW 풀어쓰기)
- F3: 프로젝트 단위형 (스킬의 양식 B)
- F4: 성과 강조 역순형 (스킬의 양식 C, 성과→HOW)
- F5: AR/TAR 한 줄 요약형 (스킬의 성과 기술 변형)
- F6: 인턴/저연차 정성형 (스킬의 양식 D)

**직무 분류:**
- 루틴직무: 영업, 재무회계, 인사, 생산관리, MD바잉, 품질관리, 물류SCM
- 프로젝트직무: 마케팅, IT개발, 제약영업, 홍보

**후보 양식 매핑 규칙:**

| 양식 | 적합 직무 | 적합 연차 |
|---|---|---|
| F1 | 모든 직무 | 모든 연차 |
| F2 | 루틴직무만 | 1~3년차, 4~6년차 |
| F3 | 프로젝트직무만 | 1~3년차, 4~6년차 |
| F4 | 모든 직무 | 1~3년차, 4~6년차 |
| F5 | 모든 직무 | 모든 연차 |
| F6 | 모든 직무 | **6개월~1년차 전용** |

직무·연차에 맞는 후보 양식 풀에서 2개를 무작위 선택. 단, 직전 5개 history 항목에서 같은 (직무·연차) 페어로 등장한 적 있는 양식 페어 조합과 동일한 페어는 회피(가능한 경우만, 후보가 부족하면 무시).

### STEP 6: 가상 인물 1명 설정

각 직무에 대해 가상 인물 1명을 만들고, 그 인물의 동일한 경력 정보를 양식 A·B 두 가지로 재구성한다.

**가상 인물 생성 규칙:**
- 이름: 가나다순 한글 이름 (예: 김민수, 이서연 등) — 매번 다르게
- 회사명: 실제 기업명 사용 금지. "○○패션", "OO식품", "△△제약" 등 마스킹 표기
- 학력: "OO대학교"로 통일
- 근무기간: 연차에 맞게 역산하여 산출 (오늘 날짜 기준)
- 업무 내용·성과: 스킬의 직무별 성과 표현 레퍼런스와 전략 패턴을 활용해 현실적으로 작성
- 이직 상황에 따라 이전 직장과 지원 직장을 다르게 설정 (예: "사기업→공기업"이면 이전 회사는 사기업, 지원처는 공기업으로 설정하여 "지원하려는 직무" 포지션 명시)

**양식 A → 양식 B 재구성 시 주의:**
- 직무, 연차, 회사명, 근무기간, 핵심 업무, 핵심 성과 수치는 동일하게 유지
- 양식 구조와 표현 방식만 다르게 변형
- 예: F2(업무단위 상세형)에서 카테고리별로 풀어 쓴 내용을 → F4(성과 강조 역순형)에서는 성과 먼저 제시 후 HOW 하위 항목으로 재배치

### STEP 7: career-description-writer 스킬 호출하여 docx 6개 생성

스킬 위치: `/mnt/skills/user/career-description-writer/SKILL.md` (또는 사용자의 클로드코드 환경에 맞는 경로)

각 직무 × 각 양식 = 6번 호출. 호출 시 다음 정보를 입력으로 전달:
- 직무
- 연차
- 이직 상황
- 가상 인물 정보 (이름, 회사명, 근무기간, 업무 내용, 성과)
- **출력 양식 지정** (F1~F6 중 어느 것)

스킬 내부의 docx 생성 규칙(11번 항목)을 준수.

### STEP 8: 파일 저장

저장 경로: `outputs/{오늘날짜}/경력기술서_{직무}_{양식코드}.docx`

예시:
```
outputs/2026-04-29/경력기술서_마케팅_F3.docx
outputs/2026-04-29/경력기술서_마케팅_F4.docx
outputs/2026-04-29/경력기술서_재무회계_F2.docx
outputs/2026-04-29/경력기술서_재무회계_F5.docx
outputs/2026-04-29/경력기술서_생산관리_F1.docx
outputs/2026-04-29/경력기술서_생산관리_F6.docx
```

### STEP 9: history.json 업데이트

오늘 작업 결과를 `history` 배열에 append:

```json
{
  "date": "2026-04-29",
  "cycle": 1,
  "outputs": [
    {
      "job": "마케팅",
      "year": "1~3년차",
      "situation": "같은직무+업계이동",
      "formats": ["F3", "F4"],
      "person_name": "김민수",
      "files": [
        "outputs/2026-04-29/경력기술서_마케팅_F3.docx",
        "outputs/2026-04-29/경력기술서_마케팅_F4.docx"
      ]
    },
    {
      "job": "재무회계",
      "year": "4~6년차",
      "situation": "사기업→공기업",
      "formats": ["F2", "F5"],
      "person_name": "이서연",
      "files": [...]
    },
    {
      "job": "생산관리",
      "year": "6개월~1년차",
      "situation": "같은직무이동",
      "formats": ["F1", "F6"],
      "person_name": "박지훈",
      "files": [...]
    }
  ]
}
```

`current_cycle`, `remaining_in_cycle`, `used_combos`도 함께 업데이트.

### STEP 10: 깃허브 커밋 & 푸시

레포 루트에서 다음 명령 실행:

```bash
git add outputs/{오늘날짜}/ history.json
git commit -m "daily: {오늘날짜} 경력기술서 6편 생성 (cycle {N})"
git push
```

푸시 실패 시 사용자에게 에러 메시지 그대로 전달 (인증 문제일 수 있음).

### STEP 11: 결과 보고

사용자에게 다음 형식으로 요약 보고:

```

    → F2 업무단위 상세형 / F5 AR/TAR 한 줄형
```

## 절대 규칙

1. **history.json을 반드시 먼저 읽고 시작**한다. 안 읽고 무작위로 뽑으면 중복 발생.
2. **used_combos에 있는 (직무|연차|상황) 조합은 절대 다시 뽑지 않는다.**
3. **remaining_in_cycle에 없는 직무는 이번 사이클에 절대 뽑지 않는다.** (사이클 균등 분배 원칙)
4. **가상 인물의 회사명에 실제 기업명을 사용하지 않는다.** (○○패션, △△제약 등 마스킹)
5. **양식 A와 B는 같은 가상 인물·같은 경력의 다른 표현**이지, 다른 사람이 아니다.
6. **스킬의 8가지 원칙(최채용솔로가정부)과 연차별 현실적 성과 수준을 반드시 준수**한다. 1년차에게 "팀 전체 전략 수립" 같은 과장 금지.
7. **history.json 업데이트는 docx 6개 생성 완료 후 한 번에**. 중간에 실패하면 history는 업데이트하지 않는다.

---

## 트러블슈팅

- **"used_combos가 너무 길어서 후보가 없음"**: 해당 직무의 used_combos만 비우고 재시도 (약 55일 후 발생 가능)
- **"git push 실패"**: 사용자에게 인증 설정 안내 (PAT 토큰 또는 SSH 키)
- **"오늘 이미 생성됨"**: STEP 2 절차대로 사용자에게 확인 후 진행
